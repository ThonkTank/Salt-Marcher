# July 2026 Design Notes

## Catalog workspace migration

The consolidated Catalog currently combines seven provider-owned reference
surfaces, but it still behaves like two UIs joined together: Monster owns the
shell controls slot while the other sections place controls inside their main
content. Scene and Catalog also issue unrelated queries into one mutable
creature-catalog model, so activating Scene can replace Catalog rows and make
the Monster table rebuild and flicker.

The target keeps Catalog presentation-only and provider APIs authoritative.
Creature browsing becomes a request/response query API; Scene and World
Planner consume a separately refreshed, immutable, revisioned creature
reference index. Catalog owns a typed section coordinator whose persistent
sections each expose controls, content, and idempotent activation hooks. A
fixed section rail lives at the top of the shell controls slot, the selected
section's controls always live below it, and only the selected section's
content appears in the main slot. Monster columns are configured once and row
updates preserve selection by creature id.

The visual direction is a compact GM reference index using the existing
SaltMarcher dark palette and typography. Its signature is one quiet brass-like
accent rail for the selected reference section; surrounding controls remain
flat and dense, with no transition animation. The rail wraps in the narrow
controls column and keeps the same order as the main reference vocabulary:

```text
[ Monster ][ Items ][ Encounter ][ NPCs ]
[ Fraktionen ][ Orte ][ Encounter-Tabellen ]
------------------------------------------------
active section controls

                    active section content
```

Alternatives rejected are keeping the nested `TabPane`, publishing every
consumer query through one global page model, and copying creature truth into
Catalog. They retain the inconsistent control placement, cross-feature query
collision, or introduce a second data owner. The migration is atomic for
internal Java APIs; persisted data and accepted observable behavior do not
change. `architectureTest`, catalog/creature/scene UI and application tests,
and the final `check` own automated proof; visual acceptance remains owner
manual testing.
