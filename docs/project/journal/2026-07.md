# July 2026 Journal

## 2026-07-04 governance-simplification - Replace procedural governance with outcome checks

Problem: Workflow instructions had accumulated procedural detail that obscured the outcomes the repository must protect.
Target: Keep short role, risk, ownership, and verification rules while moving durable technical policy to its owning documents and executable checks.
Alternatives rejected: Retaining the procedural inventory was rejected because it duplicated Git history and made instructions harder to follow.
Scope: Agent and project governance only; product behavior and required outcome checks remain unchanged.
Done when: The governing instructions identify owners, required proof, and prohibited outcomes without narrating routine execution.

## 2026-07-04 documentation-upkeep-steward - Make documentation stewardship ownership-based

Problem: Broad documentation cleanup responsibilities encouraged parallel inventories and status documents without clear authority.
Target: Assign upkeep to the owner of each documented surface and use the documentation index only for discovery and placement.
Alternatives rejected: A separate documentation-debt ledger was rejected because it would compete with owner documents and GitHub issues.
Scope: Project documentation placement and maintenance, not feature acceptance.
Done when: Every retained document has an owner route and findings are repaired in scope or tracked in GitHub.

## 2026-07-05 target-operating-model - Define the repository operating model around owned outcomes

Problem: The repository mixed stable policy, temporary delivery state, and historical reporting in overlapping document families.
Target: Keep stable policy in owner documents and ADRs, temporary sequencing in delivery manifests, and history in Git.
Alternatives rejected: Keeping health reports and migration ledgers as standing governance was rejected because their authority drifted after delivery.
Scope: Repository governance and delivery-document lifecycle.
Done when: Each document class has one role and completed delivery material is removed or explicitly historical.

## 2026-07-05 phase3-honest-instruments-design - Compare protected verification assets with their base versions

Problem: A branch could weaken the checks that were supposed to validate that same branch.
Target: Evaluate protected verification ownership and expected behavior against the protected base, while running the candidate implementation as the subject under test.
Alternatives rejected: Trusting candidate-owned policy or reviewer code was rejected because both can be changed by the candidate.
Scope: Self-protecting verification infrastructure and its base-reference gates.
Done when: Candidate changes cannot redefine the policy, ownership map, or expected verdict used to judge them.

## 2026-07-06 adr-0003-attack-incident - Targeted Warden bypass attempts exposed self-verification risk

Incident: Pull requests #386, #387, and #388 attempted targeted Warden ownership bypass, an always-pass reviewer, and a reduced harness map.
Cause: Candidate-controlled verification assets could previously participate in deciding whether their own changes were acceptable.
Prevention: Base-reference gates compare protected policy, reviewer behavior, and ownership expectations with the protected base before accepting candidate results.
Scope: The three pull requests were rejected and closed unmerged; this entry records the attack classes, not their execution transcript.
Done when: Equivalent changes fail even when the candidate modifies the local verification implementation.

## 2026-07-13 documentation-enforcement-drift - Documentation enforcement had diverged from its owner policy

Incident: Documentation checks encoded stale placement and inventory assumptions after the owner policy had changed.
Cause: Enforcement rules were maintained as an independent description instead of an executable projection of the owner document.
Prevention: Documentation enforcement now checks current owner-defined placement and hygiene outcomes, with focused selftests for the mapped categories.
Scope: Documentation enforcement and its metadata, not product documentation content.
Done when: The owner policy and checker reject the same invalid states without requiring parallel status documents.

## 2026-07-14 verification-headless-design - Make desktop verification explicitly headless

Problem: Desktop behavior proof could depend on an available display and therefore vary by execution environment.
Target: Route automated desktop verification through the repository's headless Monocle configuration.
Alternatives rejected: Depending on a real display was rejected because it makes local and CI proof non-reproducible.
Scope: Automated verification only; owner manual acceptance remains a separate activity.
Done when: Required automated desktop checks run without a display server.

## 2026-07-14 verification-parallel-settings - Isolate parallel Gradle verification settings

Problem: Concurrent verification workers could share mutable Gradle settings and interfere with one another.
Target: Give each verification invocation an explicit isolation identity and derive worker-local mutable state from it.
Alternatives rejected: Serializing all verification was rejected because isolation can preserve parallel throughput without shared-state ambiguity.
Scope: Gradle verification process state, not production runtime concurrency.
Done when: Parallel verification invocations do not share mutable settings state.

## 2026-07-14 verification-parallel-isolation - Separate parallel verification outputs and locks

Problem: Parallel verification could contend for generated outputs, locks, and transient evidence even after settings isolation.
Target: Partition mutable build outputs and coordination files by verification identity while keeping immutable inputs shared.
Alternatives rejected: Treating contention as transient was rejected because intermittent proof is not reliable proof.
Scope: Verification infrastructure outputs and coordination files.
Done when: Parallel runs cannot overwrite or lock each other's mutable evidence.

## 2026-07-15 governance-cleanup - Remove obsolete health, hook, and migration governance

Problem: Completed migrations, project-health machinery, and repository-managed local hooks remained discoverable as if they were current authority.
Target: Delete obsolete governance surfaces, keep durable decisions in ADRs, and make current delivery ownership explicit.
Alternatives rejected: Archiving obsolete machinery in place was rejected because discoverability would continue to imply operability.
Scope: Governance documents, checks, skills, tracked hook installation, and their references; binding outcome checks remain.
Done when: No active instruction or index routes agents through deleted governance, health, or hook mechanisms.

## 2026-07-15 delivery-ownership-unblock - Use a current delivery manifest for remaining verification work

Problem: The historical harness-modernization roadmap still appeared capable of activating unfinished work after its proposal had been rejected.
Target: Mark the roadmap non-operative and make the verification-harness replacement manifest the sole current sequencing authority.
Alternatives rejected: Retaining T5 as an activatable roadmap target was rejected because it would revive a superseded proposal.
Scope: Delivery ownership and sequencing only; ADRs and verification owner documents retain stable policy.
Done when: The historical roadmap points to the manifest and cannot be read as authorizing T5 or any later work.
