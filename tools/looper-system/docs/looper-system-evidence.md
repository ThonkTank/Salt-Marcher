# Looper System Evidence

## Purpose

This inventory records retained proof that the local continuous autonomous
Looper System is reproducible from the repository and follows the current
`looper-system.md` contract.

## Verified Sources

- `looper-system.md`
- `tools/looper-system/bin/looper.sh`
- `tools/looper-system/prompts/task-prompt.md`
- `tools/looper-system/bin/install.sh`
- `tools/looper-system/state/` operational runtime generated by the installer

## Evidence 2026-07-07

- Syntax: `bash -n tools/looper-system/bin/looper.sh` and
  `bash -n tools/looper-system/bin/install.sh` exited `0`.
- Dry-run: `LOOPER_DRY_RUN=1 ... bash tools/looper-system/bin/looper.sh`
  exited `0` and emitted telemetry result `dry_run_ok`.
- Work-mix backpressure: enforced meta-only exhaustion uses
  `result=work_mix_skipped`; old `result=no_work` telemetry is normalized
  before writeback and is retained only as historical input.
- Queue completion: `maybe_archive_queue_task` archived a queue item only after
  the matching `.done` file existed and telemetry used `result=queue_done`.
- Documentation: `./gradlew checkDocumentationEnforcement --console=plain`
  completed with `BUILD SUCCESSFUL`.
- Installed parity: `cmp -s` returned `0` for repo versus installed looper,
  task prompt, and user-systemd unit.
- Live service: `tools/looper-system/bin/install.sh` installed from the repo
  and restarted `saltmarcher-looper.service`; `systemctl --user status`
  reported the service `active (running)`.
- Live looper behavior: after the repo install, session `52` ran with
  `LOOPER_QUOTA_MODE=merge_pressure,r1_pressure`, selected existing green PR
  `#350`, enabled auto-merge, observed all required checks passing, and then
  entered the normal `600` second session pause.

## Evidence 2026-07-07 Repo-Local Looper Move

- Runtime root: `tools/looper-system/state/` owns the working clone, queue,
  queue archive, telemetry, logs, reports, locks, pause sentinels, and
  daily/monthly run sentinels.
- Installer migration: `tools/looper-system/bin/install.sh` stops the user
  service, moves legacy Looper state into `tools/looper-system/state/`, removes
  `~/.local/bin/saltmarcher-autodev.sh`,
  `~/.local/share/saltmarcher-autodev`, and
  `~/.local/state/saltmarcher/autodev`, writes the active unit under
  `tools/looper-system/state/systemd/`, and links that unit into user systemd.
- Manifest readback: `tools/looper-system/scripts/looper_readback.py` checks
  repo-local Looper file hashes, the active repo-local systemd unit, and
  explicit absence of the legacy Home looper paths.
- Looper PR provenance: the wrapper adds `source:looper` to every PR with
  telemetry `pr_number`, and the July telemetry-backed PR set was labeled
  retroactively.

## Proof Boundary

This evidence qualifies the local Looper packaging, prompt contract, legacy
result normalization, repo-local runtime placement, legacy path removal, and
current service activation on this machine. It does not prove that every future
scout will find the best possible refactor target; that remains review-owned
through the PR and telemetry evidence each Looper session emits.
