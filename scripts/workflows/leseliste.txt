┌─────────────────────────────────────────────────────────────────┐
│                    LESELISTE (NICHT ÜBERSPRINGBAR)              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ⛔ STOPP - LIES DIESEN BLOCK VOLLSTÄNDIG                       │
│                                                                 │
│  WARUM DIE LESELISTE NICHT ÜBERSPRUNGEN WERDEN DARF:            │
│  ─────────────────────────────────────────────────────────────  │
│  • Ohne Kontext produzierst du Code der nicht passt             │
│  • Du weißt nicht welche Patterns/APIs bereits existieren       │
│  • Du wiederholst Fehler die andere schon gemacht haben         │
│  • Du verstehst die Architektur-Entscheidungen nicht            │
│  • Deine Lösung wird bei Review abgelehnt → verschwendete Zeit  │
│                                                                 │
│  → Die Leseliste IST der Kontext. Ohne ihn arbeitest du blind.  │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  WAS MUSS GELESEN WERDEN (in dieser Reihenfolge):               │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  A) ARCHITEKTUR-BASELINE (PFLICHT - IMMER):                     │
│     Diese Docs definieren WIE Code geschrieben werden muss:     │
│     → Conventions.md      (Code-Stil, Import-Reihenfolge)       │
│     → Error-Handling.md   (Result-Pattern, AppError)            │
│     → Events-Catalog.md   (Event-Naming, correlationId)         │
│     + Layer-spezifische Docs (falls Layer != '-')               │
│                                                                 │
│  B) FEATURE-DOCS (PFLICHT - ALLE für diesen Bereich):           │
│     Diese Docs definieren WAS gebaut werden muss:               │
│     → Alle Docs aus featureRouting für den Task-Bereich         │
│     → Du darfst KEINEN überspringen, auch wenn er "unwichtig"   │
│       erscheint - oft sind kritische Edge-Cases dort versteckt  │
│                                                                 │
│  C) SPEC-SPALTE DER TASK:                                       │
│     Spezifische Sektionen mit Anker-Referenzen:                 │
│     → Diese zeigen GENAU welche Stellen relevant sind           │
│     → Format: Doc.md#sektion → zu dieser Überschrift navigieren │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  WIE DIE LESELISTE ABGEARBEITET WERDEN MUSS:                    │
│  ─────────────────────────────────────────────────────────────  │
│                                                                 │
│  SCHRITT 1: TodoWrite mit ALLEN Docs erstellen                  │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ Beispiel-Format für TodoWrite:                              ││
│  │                                                             ││
│  │ { content: "Conventions.md lesen",                          ││
│  │   status: "pending",                                        ││
│  │   activeForm: "Lese Conventions.md" }                       ││
│  │                                                             ││
│  │ { content: "Error-Handling.md lesen",                       ││
│  │   status: "pending",                                        ││
│  │   activeForm: "Lese Error-Handling.md" }                    ││
│  │                                                             ││
│  │ ... (für JEDES Doc aus A, B und C)                          ││
│  └─────────────────────────────────────────────────────────────┘│
│                                                                 │
│  SCHRITT 2: Jeden Doc einzeln lesen                             │
│  → Read-Tool verwenden                                          │
│  → Bei Anker-Referenzen: Zur Sektion navigieren                 │
│  → Relevante Patterns/APIs/Types notieren                       │
│                                                                 │
│  SCHRITT 3: Nach JEDEM Doc → Todo als "completed" markieren     │
│  → Nicht alle auf einmal, sondern SOFORT nach dem Lesen         │
│                                                                 │
│  SCHRITT 4: ERST wenn ALLE Todos completed → zum Workflow       │
│                                                                 │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ⛔ VERBOTEN:                                                   │
│  ─────────────────────────────────────────────────────────────  │
│  ✗ Leseliste überspringen ("ich kenne das schon")               │
│  ✗ Nur manche Docs lesen ("die anderen sind nicht relevant")    │
│  ✗ Docs nur überfliegen (du musst sie VERSTEHEN)                │
│  ✗ TodoWrite nicht erstellen                                    │
│  ✗ Zum Workflow gehen bevor alle Todos completed sind           │
│  ✗ Todos als "completed" markieren ohne zu lesen                │
│                                                                 │
│  → Jede Verletzung führt zu Code der bei Review abgelehnt wird  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
